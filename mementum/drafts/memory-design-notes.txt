Your idea is exactly the right direction: do not design “memory” as one store. Design it as a memory control plane with one authoritative audit/provenance layer and multiple replaceable projections: temporal, graph, vector, keyword, profile, and procedural memory.

The key principle:

Append-only source of truth; replaceable read indexes.

That means the system of record should be your own durable ledger, probably Postgres. Graph DBs, vector DBs, OpenAI File Search, Zep/Graphiti, Mem0, etc. should be treated as indexes or memory services, not the only truth. This gives you audit, replay, attribution, versioning, and future pluggability.

Core data model: source, assertion, derived fact

The most important modeling choice is to separate source observations from fact assertions.

1. Source observation

This is what a source actually said or produced.

Examples:

tool_result: GTAC case query returned DTC P0A80
log_extract: vehicle log contains timestamped voltage drop
user_message: technician says “battery was replaced last week”
document_chunk: service bulletin paragraph

Store it immutably.

Fields:

source_observation_id
source_type              -- user_message, tool_result, document_chunk, log_line, model_output
source_uri               -- exact document/tool/run/message reference
raw_payload              -- JSON/text
observed_at              -- when event happened, if known
recorded_at              -- when system recorded it
case_id
conversation_id
turn_id
run_id
tool_call_id
hash
2. Fact assertion

This is the system’s interpretation.

Example:

Vehicle X has DTC P0A80
Software version Y was installed before symptom Z
Repair action A resolved symptom B
Cause C is ruled out

Fields:

fact_assertion_id
subject
predicate
object
fact_type
status                  -- proposed, observed, inferred, confirmed, contradicted, retracted, superseded
confidence
valid_from              -- when true in the world
valid_to
recorded_at             -- when system learned it
retracted_at
supersedes_assertion_id
derived_from[]          -- source observations or prior facts
attribution             -- source/tool/model/prompt/version
3. Canonical current fact

This should usually be a view, not a mutable row.

current_facts = latest non-retracted assertions
                where valid_to is null
                and status in ('observed', 'inferred', 'confirmed')

This gives you updatable memory without losing history.

Use bitemporal modeling

You need two timelines:

Valid time: when the fact was true in the real world.
Transaction/record time: when your system learned or changed the fact.

Example:

June 1: vehicle had software version A
June 10: vehicle updated to software version B
June 15: agent learns about the June 10 update

If you only store created_at, you lose the difference between when it happened and when you learned it.

For your diagnostic system, bitemporal facts are essential because vehicle state, software versions, repairs, symptoms, and case conclusions change over time.

Graphiti/Zep is very relevant here: Graphiti describes itself as a temporal knowledge graph that tracks entities, relationships, facts, changing relationships, historical context, temporal metadata, provenance, and fact invalidation.

Write path: treat memory writes as controlled events

A strong write path should look like this:

raw input/tool result
  → observation event
  → candidate fact extraction
  → validation/deduplication
  → provenance links
  → optional human/policy approval
  → fact assertion
  → projection updates: graph, vector, keyword, temporal views

Do not let the LLM directly mutate the current memory state. Let it propose memory operations:

{
  "operation": "assert_fact",
  "subject": "vehicle:VIN123",
  "predicate": "has_dtc",
  "object": "P0A80",
  "confidence": 0.82,
  "derived_from": ["tool_result:abc", "log_line:def"]
}

Then deterministic code validates and commits.

This aligns with recent memory-security research: long-term memory introduces persistence, propagation, and rollback risks, so governance has to be anchored in storage-time provenance, versioning, retention policy, and auditable control rather than patched on at retrieval time.

Read path: use a retrieval router, not one search call

At query time, use a memory retrieval router:

User asks diagnostic question
  → classify retrieval need
  → fetch current case state
  → fetch active temporal facts
  → traverse graph for related entities
  → retrieve similar prior cases/chunks
  → retrieve exact keyword matches
  → rerank / merge / dedupe
  → produce evidence bundle
  → agent answers with citations

For example:

Query type	Retrieval strategy
“What changed before the fault?”	temporal facts + event timeline
“Have we seen this DTC before?”	vector + keyword + graph
“Which ECU/software version is implicated?”	graph traversal + temporal validity
“Why did the agent conclude X?”	derivation DAG + source observations
“What is the latest confirmed diagnosis?”	current fact view
“Show full audit trail”	ledger timeline

OpenAI File Search is a good reference for managed hybrid retrieval: it rewrites queries, breaks complex queries into parallel searches, runs keyword and semantic search, and reranks results before generating an answer. But for your system, I would not make it the authoritative store because you need deterministic metadata filters, source/fact lineage, and structured diagnostic facts.

Do not overwrite important memory rows. Insert correction, supersession, or retraction records.

2. Vector search

Start simple:

Option A: pgvector

Best when you want one DB, simple ops, strong attribution joins, and moderate scale. pgvector supports exact and approximate nearest-neighbor search, multiple vector types, several distance functions, and works with any Postgres client. It also supports HNSW indexes, where higher ef_search improves recall at the cost of speed.

Use pgvector for:

prior case summaries
document chunks
tool-result summaries
diagnostic notes
fact assertion text
“lesson learned” entries

Option B: Qdrant

Good when you need stronger vector infra, named vectors, hybrid/multi-stage retrieval, reranking, grouping, and lower operational coupling to Postgres. Qdrant’s Query API supports combining multiple queries and multi-stage retrieval.

Option C: Weaviate

Good for managed hybrid search and schema-oriented object retrieval. Weaviate hybrid search combines vector search and BM25F keyword search with configurable fusion and weights.

Option D: OpenAI File Search

Good for fast managed document RAG. It automatically parses, chunks, embeds, stores, uses keyword + semantic search, and reranks. Use it for prototypes or non-sensitive document search, but keep your canonical facts and attribution in your own DB.

My recommendation for you: Postgres + pgvector first; design an interface so Qdrant or Weaviate can replace it later.

3. Graph memory

For your use case, graph memory is not optional. Vehicle diagnosis is naturally relational:

case → vehicle → ECU → software version → DTC → symptom → test result → repair → outcome

Good options:

Graphiti / Zep

Best fit for temporal agent memory. Graphiti builds dynamic temporal context graphs from structured and unstructured data, supports changing relationships, historical context, provenance, entity/edge extraction, custom entity types, and hybrid search over time/full-text/semantic/graph algorithms.

Use Graphiti if you want:

temporal knowledge graph
fact invalidation
evolving relationships
provenance-aware graph episodes
local open-source graph memory

Use Zep if you want managed scale/governance on top of Graphiti. The docs position Zep as enterprise-scale agent memory with managed extraction, retrieval, storage, governance, and low-latency serving.

Neo4j

Best if you want mature graph database operations, Cypher, visualization, constraints, enterprise features, and long-term maintainability. It is not agent-memory-specific, so you implement temporal validity/provenance yourself.

LlamaIndex PropertyGraphIndex

Good for building GraphRAG pipelines and experimenting with graph extraction. LlamaIndex’s PropertyGraphIndex constructs and queries property graphs; it supports graph stores such as Neo4j, Nebula, TiDB, and FalkorDB, and can combine a graph store with an external vector DB like Qdrant.

My recommendation: start with your own fact tables + optional Graphiti projection. Do not put your only copy of facts in Graphiti. Treat Graphiti/Neo4j as a read/query projection.

7. Background consolidation

Do memory extraction outside the critical response path when possible.

Letta’s sleep-time agent pattern is a good reference: a background agent shares memory with the primary agent and asynchronously updates memory blocks from conversation history or data sources. Letta also describes memory blocks as labeled context-window sections with size limits that can be updated, retrieved, deleted, and shared across agents.

For your implementation, create a memory_consolidator job:

trigger: after run, after case close, nightly, or manual review
input: new ledger events since last checkpoint
output:
  - candidate facts
  - candidate graph edges
  - case summary
  - lesson learned
  - retractions/supersessions
  - retrieval index updates

Keep two modes:

hot path: only store raw observations and minimal candidate facts
background: dedupe, merge, summarize, promote, invalidate

Recent surveys describe modern agent memory as a write–manage–read loop, with open problems around consolidation, contradiction handling, latency, privacy, and learned forgetting.

Interfaces for pluggability

Define ports/interfaces first; each backend is an adapter.

class MemoryLedger:
    def append_event(self, event: MemoryEvent) -> EventId: ...
    def get_timeline(self, scope: Scope) -> list[MemoryEvent]: ...

class FactStore:
    def assert_fact(self, assertion: FactAssertion) -> FactAssertionId: ...
    def retract_fact(self, assertion_id: str, reason: str) -> None: ...
    def get_current_facts(self, scope: Scope) -> list[FactAssertion]: ...
    def get_facts_as_of(self, scope: Scope, as_of: datetime) -> list[FactAssertion]: ...

class ProvenanceStore:
    def link_derivation(self, child_id: str, parent_ids: list[str], method: str) -> None: ...
    def explain(self, object_id: str) -> ProvenanceGraph: ...

class VectorIndex:
    def upsert(self, item: IndexedMemory) -> None: ...
    def search(self, query: str, filters: dict, k: int) -> list[SearchHit]: ...

class GraphIndex:
    def upsert_entity(self, entity: Entity) -> None: ...
    def upsert_relation(self, relation: Relation) -> None: ...
    def traverse(self, query: GraphQuery) -> list[GraphHit]: ...

class MemoryRetriever:
    def retrieve(self, request: RetrievalRequest) -> EvidenceBundle: ...

The application should depend on these interfaces, not on Qdrant/Neo4j/Zep/OpenAI APIs directly.

Update and invalidation strategy

Avoid “update in place” for meaningful facts.

Use these operations:

Operation	Meaning
assert_fact	add new claim
confirm_fact	mark claim as validated
supersede_fact	replace old claim with newer claim
contradict_fact	preserve both claims but mark conflict
retract_fact	withdraw a claim
close_validity	fact was true but no longer true
merge_entity	entity resolution says two nodes are same
split_entity	undo bad entity merge

Important: contradiction is not deletion. In diagnostic systems, contradictory evidence is valuable.

Retrieval evidence bundle

Make every memory retrieval return structured evidence, not just text.

{
  "query": "why do we think ECU software caused the issue?",
  "hits": [
    {
      "object_type": "fact_assertion",
      "object_id": "...",
      "score": 0.91,
      "retrieval_path": ["graph", "temporal"],
      "summary": "Software version B was installed before symptom onset.",
      "source_observations": ["...", "..."],
      "valid_from": "2026-06-10T00:00:00Z",
      "status": "confirmed"
    }
  ]
}

Then the agent response can cite evidence instead of vaguely saying “memory says.”

What I would avoid

Avoid these common mistakes:

One vector DB as memory
You will lose update semantics, temporal reasoning, exact code matching, and clear provenance.
Summaries as the only long-term memory
Summaries are useful, but they are lossy and hard to audit.
LLM directly mutating canonical facts
Let the LLM propose; let deterministic code validate and commit.
Current-state-only storage
You need history, invalidation, and as-of queries.
Graph without provenance
A graph edge without source attribution becomes untrustworthy quickly.
Mixing case-local facts and global knowledge
Case facts can be wrong or incomplete. Promote them to global diagnostic knowledge only after review or policy gates.
Best-fit design for your diagnostic agent

I would use four durable memory scopes:

case_memory
  Facts and evidence for one HAR/GTAC case.

vehicle_memory
  Vehicle-specific history: VIN, ECU versions, prior repairs, prior DTCs.

fleet_memory
  Cross-case patterns, known issues, issue clusters, recurring fixes.

agent_memory
  Procedural lessons: tool usage, planning mistakes, evaluation failures.

And four confidence levels:

observed     -- directly from source/tool/log
inferred     -- model or rule-derived
confirmed    -- validated by source or human
deprecated   -- superseded/retracted/invalid

Promotion policy:

case observation
  → case fact
  → case conclusion
  → reviewed lesson
  → fleet/global diagnostic knowledge

That gives you safety and auditability.

Final recommendation

Build your system around this invariant:

Every memory must be explainable by a timeline and a provenance graph.

Use Postgres as the immutable ledger and temporal fact store. Add pgvector first for semantic retrieval. Add Graphiti or Neo4j as a graph projection. Add Qdrant/Weaviate/OpenAI File Search later only behind a VectorIndex interface. Use background consolidation to promote raw observations into durable facts and lessons. Keep case-local, vehicle-local, fleet-global, and agent-procedural memory separate.

That design will be much more durable than choosing any single “agent memory product,” and it will let you plug in better graph/vector/memory tools as the ecosystem evolves.
