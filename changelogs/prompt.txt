Compare the current branch against the `main` branch.

Create a concise reviewer-focused changelog that explains only the information that helps a human reviewer understand the change. Avoid repeating facts that are obvious from reading the diff, such as simple line deletions, renames, or mechanical edits, unless they affect behavior, design, or review context.

Include:

1. A brief feature-level summary of the branch.
2. The rationale or likely motivation for meaningful changes.
3. A file-by-file explanation of changed files, focused on why each file changed and what reviewers should pay attention to.
4. A line-level or section-level interpretation only where the code change is non-obvious, behaviorally important, or useful for review.

Keep the report concise. Prefer higher-level explanations over restating the diff. Do not include filler, obvious observations, or overly verbose descriptions.

Write the final report to:

`changelogs/YYYYMMDD-feature-summary-changes.md`

Replace `YYYYMMDD` with today’s date.

Do not modify application code. Only create or update the changelog file.
