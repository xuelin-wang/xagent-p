Design Document: Pluggable Common LLM Wrapper for xagent-p
1. Purpose
Implement a reusable, provider-pluggable LLM wrapper inside the existing xagent-p repository.
Initial supported providers:
openaianthropic
Initial supported model families:
OpenAI:  gpt-5.5  gpt-5.4  gpt-5.4-miniOpenAI embeddings:  text-embedding-3-small  text-embedding-3-largeAnthropic:  claude-opus-4-7  claude-sonnet-4-6  claude-haiku-4-5-20251001
OpenAI’s current model docs recommend gpt-5.5 as the starting point for complex reasoning and coding, with smaller variants such as gpt-5.4-mini for lower-cost and lower-latency workloads. The docs also list function tools, web search, file search, and computer use as supported tools for recent OpenAI models.  Anthropic’s current model docs list Claude Opus 4.7, Claude Sonnet 4.6, and Claude Haiku 4.5 as latest models, with Claude API IDs claude-opus-4-7, claude-sonnet-4-6, and claude-haiku-4-5-20251001. 
The wrapper must support:
1. Text generation2. Structured / formatted output3. Embeddings, OpenAI only initially4. File upload and file references5. App-hosted tool calls6. Provider-hosted tool calls7. Batch requests8. Retry, timeout, and error normalization9. Provider plugin architecture
The design must follow the existing Polylith convention used in this repo: namespace xagent, loose Polylith layout, top-level components/, bases/, projects/, and development/. The repo’s workspace.toml declares namespace xagent and loose structure, and the root pyproject.toml builds packages from components/xagent and bases/xagent.  

2. Non-goals for Initial Implementation
Do not implement these initially:
1. Gemini2. Vertex AI Gemini3. DeepSeek4. Azure OpenAI5. Bedrock Anthropic6. Streaming7. Realtime APIs8. Audio input/output9. Image generation10. Fine-tuning11. Automatic vector store lifecycle management12. Full MCP client/server implementation13. Automatic execution of app-hosted tools inside the low-level provider wrapper
The design must still preserve a plugin architecture so these providers/features can be added later.

3. Key Design Rule
Normalize the common surface area, but do not hide important provider differences.
Use this rule throughout the implementation:
Normalize request/response shape.Preserve raw provider responses.Capability-gate provider-specific features.Fail loudly when unsupported.Never silently emulate one provider’s behavior with another provider’s behavior.
Examples:
- Do not use OpenAI embeddings automatically for Anthropic requests.- Do not emulate provider-hosted web search with an app-hosted search tool unless explicitly requested by the caller.- Do not assume OpenAI and Anthropic file APIs have the same semantics.- Do not assume provider-hosted tools and app-hosted tools follow the same execution model.

8. Retry Component
8.1 xagent.llm_retry
Files:
retry.pytimeout.py
Retry these by default:
HTTP 408HTTP 409, only if provider marks transientHTTP 429HTTP 500HTTP 502HTTP 503HTTP 504network timeoutconnection resettemporary DNS failure
Do not retry these by default:
HTTP 400HTTP 401HTTP 403HTTP 404schema validation failure after configured validation retriesapp tool execution failureside-effecting operation without idempotency key
Implementation requirements:
1. Respect Retry-After when present.2. Otherwise use exponential backoff.3. Add jitter when configured.4. Stop at max_attempts.5. Preserve provider request ID in normalized errors when available.6. Never log API keys.

10. Tools Component
10.1 xagent.llm_tools
Files:
app_tools.pyprovider_tools.pytool_loop.pyvalidation.py
There are two separate tool baskets:
app_tools:  Tools the application executes.provider_tools:  Tools the model provider executes.
Anthropic explicitly documents this distinction as client tools versus server tools: client tools are executed by your system, while server tools are executed by Anthropic.  OpenAI similarly distinguishes custom function tools from hosted tools such as web search, file search, code interpreter, image generation, and computer use. 
App-hosted tools

Tool choice
class ToolChoice(BaseModel):    mode: Literal["auto", "none", "required"] = "auto"    tool_name: str | None = None
Tool execution policy
The low-level provider wrappers must not execute app-hosted tools automatically. They should only return normalized AppToolCall objects.
Add an optional helper:
class AppToolExecutor(Protocol):    async def __call__(self, arguments: dict[str, Any]) -> Any:        ...async def run_app_tool_loop(    provider: LLMProvider,    request: GenerateRequest,    executors: dict[str, AppToolExecutor],    max_rounds: int = 5,) -> GenerateResponse:    ...
Tool loop behavior:
1. Send request.2. If response has no app_tool_calls, return response.3. Validate arguments against input_schema.4. Execute app tools.5. Append tool result messages.6. Repeat until final answer or max_rounds.7. Never blindly retry side-effecting tool execution.

OpenAI documents text-embedding-3-small and text-embedding-3-large as its newer third-generation embedding models, with default vector sizes of 1536 and 3072 respectively and a configurable dimensions parameter. 
Anthropic embeddings should raise:
UnsupportedCapabilityError(    LLMErrorPayload(        provider="anthropic",        operation="embed",        message="Anthropic embeddings are not supported by this wrapper.",        retryable=False,    ))

17. Capability Checks
Every provider must capability-check before sending a request.
def assert_capability(    capabilities: ModelCapabilities,    capability: Capability,) -> None:    if capability not in capabilities.capabilities:        raise UnsupportedCapabilityError(            LLMErrorPayload(                provider=capabilities.provider,                model=capabilities.model,                operation="capability_check",                message=f"Capability {capability} is not supported.",                retryable=False,            )        )
Required checks:
if request.app_tools:    assert_capability(caps, Capability.APP_TOOL_CALLS)if request.provider_tools:    assert_capability(caps, Capability.PROVIDER_HOSTED_TOOLS)if request.app_tools and request.provider_tools:    assert_capability(caps, Capability.MIXED_APP_AND_PROVIDER_TOOLS)if request.files:    assert_capability(caps, Capability.FILE_INPUT)if request.response_format and request.response_format.type != ResponseFormatType.TEXT:    assert_capability(caps, Capability.STRUCTURED_OUTPUT)

21. Import Rules
Follow these rules strictly:
Core components must not import provider components.Provider components may import:  xagent.llm_contracts  xagent.llm_config  xagent.llm_retry  xagent.llm_files  xagent.llm_tools  xagent.llm_structured  xagent.llm_batch  xagent.llm_registryllm_registry.provider_protocol must not import provider implementations.llm_registry.factory may import built-in provider implementations only inside default_registry().Bases may import components.Tests should mirror component paths under test/components/xagent.
Bad:
# Do not do this inside llm_contractsfrom xagent.llm_provider_openai import OpenAIProvider
Good:
# Provider imports shared contractsfrom xagent.llm_contracts import GenerateRequest, GenerateResponse

24. Future Provider Extension Pattern
To add a future provider such as Gemini, Vertex Gemini, DeepSeek, Azure OpenAI, or Bedrock Anthropic:
1. Create a new component:   components/xagent/llm_provider_<provider_name>/2. Implement LLMProvider protocol.3. Add provider-specific request/response mapping.4. Register it with ProviderRegistry.5. Add tests under:   test/components/xagent/llm_provider_<provider_name>/
Do not modify shared request models unless the new provider exposes a genuinely new cross-provider concept.
